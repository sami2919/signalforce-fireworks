# GTM Context: Kana (Customer Zero AI-First Engine)

Shared context for a Kana-targeted SignalForce branch.

## Company

| Field | Details |
|---|---|
| Name | Kana |
| Category | Agentic marketing platform |

| Internal thesis | Maximize at-bats by treating the funnel like software |

## What the role actually owns

- Outbound engine: Clay, Wiza, Sales Navigator, Instantly, Valley
- System-of-record plumbing: HubSpot + zero manual data entry
- Inbound loop: SEMRush-driven keyword discovery and AI-assisted content deployment
- Paid + social feedback loop: route experiment learnings back into spend
- Customer Zero usage: use Kana internally to find audiences, write experiments, and improve discoverability

## Best story for the walk-in

The branch should prove two things:

1. You can build the internal demand engine the JD describes.
2. You understand how Kana should use its own platform inside that engine.

Do not frame this as “I rebuilt Kana.”
Frame it as:

> “You want a GTM engineer who can own sourcing, routing, experimentation, HubSpot plumbing,
> and AI-native execution as one system. I already built the operating system for that.”

## Priority signals

1. Hiring into growth, lifecycle, RevOps, or marketing ops
2. Public leadership activity about AI-first marketing, experimentation, or automation
3. Fresh funding that implies GTM expansion
4. Technical or content artifacts showing complex marketing stack usage
5. SEO / content scale pressure where AEO or answer-engine discoverability matters

## Messaging angles

- “Increase at-bats without increasing manual ops headcount”
- “Keep the current stack, replace the glue work”
- “Treat demand generation like code: test, route, measure, repeat”
- “Customer Zero: use Kana internally before asking customers to trust it”
